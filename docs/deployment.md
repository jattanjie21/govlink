# Deployment

This guide covers the two supported deployment models: a managed PaaS
(Railway is the worked example) and self-hosted Docker Compose. Both
use the same image built from the project's `Dockerfile`.

## Environment variables

Every variable is read by `govlink.config.Settings`. All names are
prefixed with `GOVLINK_`.

| Variable | Type | Default | Notes |
|---|---|---|---|
| `GOVLINK_ENV` | `development` \| `production` | `development` | Production rejects requests that lack `GOVLINK_DATABASE_URL`. |
| `GOVLINK_LOG_LEVEL` | `CRITICAL`/`ERROR`/`WARNING`/`INFO`/`DEBUG` | `INFO` | Case-insensitive. |
| `GOVLINK_DATABASE_URL` | SQLAlchemy URL | `sqlite:///./govlink.db` (dev) | Required in production. Use `postgresql+psycopg://user:pass@host:port/db` for Postgres. |
| `GOVLINK_RAW_DATA_DIR` | path | `./data/raw` | Where raw downloaded PDFs are written. The directory is created on demand. |
| `GOVLINK_API_HOST` | string | `0.0.0.0` | Bind address for the API server. |
| `GOVLINK_API_PORT` | int (1â€“65535) | `8000` | API port. |
| `GOVLINK_RATE_LIMIT_PER_MINUTE` | int (>0) | `60` | Per-IP rate limit on data routes. Meta + admin routes are unlimited. |
| `GOVLINK_CORS_ORIGINS` | comma-separated | `*` | E.g. `https://a.example,https://b.example`. |

## Deploying on Railway

1. Create a new Railway project and connect this repo. Railway
   auto-detects the `Dockerfile`.
2. Add the **Postgres** plugin from the Railway marketplace.
3. In the API service, set the following environment variables:
   - `GOVLINK_ENV=production`
   - `GOVLINK_DATABASE_URL` â€” paste the connection string Railway shows
     for the Postgres plugin. Make sure it begins with
     `postgresql+psycopg://` (Railway sometimes shows `postgresql://`
     by default â€” change the prefix).
   - `GOVLINK_RAW_DATA_DIR=/app/data/raw`
   - `GOVLINK_LOG_LEVEL=INFO`
4. Add a **release command** (Railway: Settings â†’ Deploy â†’
   "Pre-deploy command"):
   ```bash
   alembic upgrade head
   ```
   This runs migrations on every deploy before the API process is
   restarted.
5. Add a **cron job** for ingestion (Railway: Settings â†’ Cron):
   - Schedule: `0 20 * * 1-5` (8 PM UTC, weekdays â€” the CBG publishes
     by ~6 PM GMT)
   - Command: `govlink ingest exchange-rates`

## Deploying with Docker Compose (self-hosted)

The bundled `docker-compose.yml` brings up Postgres, runs Alembic
migrations once, then starts the API. Suitable for a small VPS.

```bash
git clone https://github.com/TODO/govlink.git
cd govlink

# Override the dev defaults â€” at minimum, POSTGRES_PASSWORD.
cp .env.docker .env  # or set the variables in your shell

docker compose up -d
docker compose logs -f api
```

### Scheduled ingestion via host cron

Once the stack is running, add an entry to the host's crontab:

```cron
0 20 * * 1-5 cd /opt/govlink && docker compose exec -T api govlink ingest exchange-rates >> /var/log/govlink-ingest.log 2>&1
```

The `-T` flag disables TTY allocation so the command works under cron.

### Backups

The Postgres data lives in the `pgdata` named volume. Snapshot it with
`pg_dump` from inside the container:

```bash
docker compose exec db pg_dump -U govlink govlink | gzip > "backup-$(date +%F).sql.gz"
```

Restore by piping `gunzip ... | docker compose exec -T db psql -U govlink govlink`.

The `rawdata` volume holds the original PDFs. It's safe to drop and
re-fetch â€” the ingestion pipeline is idempotent on `source_uuid`.

## Database migrations

govlink uses Alembic for schema management. The same migration files
work for SQLite (local dev) and Postgres (production).

```bash
# Apply all pending migrations
alembic upgrade head

# Generate a new migration after changing models
# (e.g., when adding a new dataset that needs a new data_<slug> table)
alembic revision --autogenerate -m "add my_new_dataset"

# Roll back the most recent migration
alembic downgrade -1
```

When adding a new dataset, the autogenerated migration usually needs a
manual review â€” verify enum CHECK constraints, naming-convention
indexes, and the per-dataset `data_<slug>` table all look right before
committing.

## Operational endpoints

- `GET /health` â€” liveness probe. Always returns 200 if the app process
  is up. Not rate-limited.
- `GET /admin/health` â€” per-dataset freshness report (last ingestion
  time, latest snapshot date, staleness flag). Suitable for Prometheus
  scrapes; not rate-limited.
